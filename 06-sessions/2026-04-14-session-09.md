### 2. `TvaAgent` TypeScript + endpoint `/agents/tva/ask`

**`src/agents/tva/TvaAgent.ts`** (~90 lignes) :
- Input : `TvaQuery { question, context? { turnover, method, sector } }`
- Enrichit la question avec le contexte si fourni
- Embed via BGE-M3 GPU (llama-server 8082)
- Qdrant search top-5
- **Re-ranking** : LTVA (0) > OLTVA (1) > Info TVA (2) > autres (10)
- Prompt `lexa-tva` avec contexte formatté
- Return : answer + citations + durationMs

**`src/routes/agents.ts`** :
- `POST /agents/tva/ask` — endpoint spécialisé
- `GET /agents` — liste des agents actifs + planifiés

## Décisions structurelles prises

1. **Lexa est indépendante** : onboarding via UID register fonctionne sans aucune dépendance à Swigs Pro. C'était la demande critique du user ("Lexa doit aussi être indépendante"). Validé end-to-end.

2. **Réimplémentation > import direct** : au lieu de partager du code entre Pro (JS) et Lexa (TS), j'ai re-écrit proprement le service UID register en TypeScript. Avantages : type safety, pas de couplage git, évolution indépendante. Coût : maintenance × 2 si BFS change son API (peu probable).

3. **`fast-xml-parser` > `xml2js`** : 2-3× plus rapide, types natifs, pas de callback hell. Migration future à prévoir dans Pro si perf devient un sujet.

4. **Hook Pro = fire-and-forget** : Pro ne dépend jamais de Lexa. Si Lexa est down, Pro continue normalement. Si Lexa est lent, Pro continue normalement. **Zero risk pour la prod user**.

5. **Flag `LEXA_ENABLED`** : opt-in explicite. Pro reste byte-for-byte identique à l'ancien comportement tant que le user n'active pas. C'est le patron à suivre pour toute future intégration Pro/Lexa.

6. **Agent TVA spécialisé > agent générique** : le test "seuils TDFN" a prouvé que le re-ranking + system prompt dédié extrait des informations plus précises (Info TVA 12 section 55) qu'un RAG naïf. Pattern à reproduire pour fiscal-PP, fiscal-PM, clôture, etc.

7. **`rankVatSources` re-ranking** : LTVA > OLTVA > Info TVA > autres. Code ~10 lignes dans TvaAgent. Peut être généralisé dans `ragQuery.ts` plus tard pour d'autres agents.
