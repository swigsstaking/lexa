### 7. Créer connecteur `POST /connectors/bank/ingest`

**Fichier** : `apps/backend/src/routes/connectors.ts`

**Schéma Zod compatible Swigs Pro BankTransaction** :
```typescript
{
  txId?: string,
  amount: number,            // positif
  currency: string (len 3),  // default CHF
  creditDebit: "CRDT" | "DBIT",
  counterpartyName?: string,
  counterpartyIban?: string,
  reference?: string,        // QR reference / SCOR
  unstructuredReference?: string,
  bookingDate: string,
  importFilename?: string,
  source?: "swigs-pro-email" | "swigs-pro-camt053" | "manual",
  userId?: uuid
}
```

**Flow pour chaque transaction** :
1. Conversion `creditDebit` → amount signé (DBIT = négatif, CRDT = positif)
2. Build `description` enrichie : `counterpartyName | reference | unstructuredReference`
3. Append `TransactionIngested` event (metadata inclut `txId`, `counterpartyName`, `importFilename`, `userId`)
4. Classify via `lexa-classifier`
5. Append `TransactionClassified` event
6. Persist `ai_decisions` audit row
7. Retourner stream_id + classification (compte, TVA, citations)

**Endpoint `GET /connectors/bank/formats`** : liste les formats supportés + planned (CAMT.053, CSV).

## Découvertes importantes

1. **`lexa-reasoning` fait gagner 5.8×** sur `/rag/ask` vs `comptable-suisse` Q8. Base qwen3.5:9b + SYSTEM dédié = plus rapide, plus ciblé, citations plus propres. La leçon : **pour chaque rôle d'agent, un Modelfile dédié** vaut mieux qu'un modèle monolithique.

2. **Le Classifier a auto-reconnu le compte 6510** (Honoraires fiduciaires) qu'on venait d'ajouter dans la KB session 07. Preuve que **le RAG sur le plan Käfer structuré est directement utile aux agents**. Quand on ajoute un compte dans Käfer YAML + ingestion → l'agent le voit immédiatement sans retraining.

3. **Encaissement client → 1100 Débiteurs** (pas 3000 Ventes) : le Classifier a correctement compris qu'un virement entrant d'un client **solde la créance existante** au lieu d'enregistrer une nouvelle vente. C'est **comptablement correct** et montre que le modèle comprend bien la logique de double-entry.

4. **Parser email Swigs Pro est très mature** (10 banques, 4 langues, 4 stratégies de reconciliation). Le réutiliser via pont HTTP est **10× plus efficace** que de refaire la roue côté Lexa.

5. **Materialized view Postgres suffit** pour le Grand Livre. Pas besoin d'un moteur CQRS sophistiqué. `REFRESH MATERIALIZED VIEW` est rapide (ms) sur nos volumes.

6. **Balance de vérification balance automatiquement** grâce à la partie double : chaque event `TransactionClassified` génère 1 ligne débit + 1 ligne crédit du même montant. Si on viole la partie double (amount_ttc différent entre débit et crédit), la view révèle immédiatement l'incohérence.
