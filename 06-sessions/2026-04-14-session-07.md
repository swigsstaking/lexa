### 5. Découverte critique : Qwen3 thinking mode

Les premiers tests retournaient `response: ""` avec `eval_count: 200` — le modèle produisait 200 tokens mais invisibles. **Le mode thinking de Qwen3 consomme les tokens dans `<think>...</think>` masqués**.

**Solution** : paramètre API `think: false` (désactive le thinking).

Test :
```bash
curl -d '{"model":"comptable-suisse-fast","prompt":"...","think":false,...}'
```

→ Réponse parfaite en prose avec citations. **Fix déployé dans `OllamaClient.ts`** avec `think: false` par défaut.

### 7. Phase 2 — BGE-M3 sur GPU via llama.cpp (gros chantier)

**Build llama.cpp** :
- Prerequis : CUDA 13.0 dans `/usr/local/cuda-13.0/bin/nvcc`
- `git clone --depth 1 github.com/ggml-org/llama.cpp`
- `cmake -B build -DGGML_CUDA=ON -DCMAKE_CUDA_ARCHITECTURES="121" -DLLAMA_CURL=OFF`
- cmake replace automatique : `121 → 121a` (spécifique GB10)
- `cmake --build build -j 20 --target llama-server llama-embedding`
- Durée : ~10 min
- Binaires : `~/llama.cpp/build/bin/llama-server` + `llama-embedding`

**Download BGE-M3 GGUF** :
- `gpustack/bge-m3-GGUF` depuis HuggingFace
- Fichier : `bge-m3-Q8_0.gguf` (606 MB)
- Path : `~/models/bge-m3/bge-m3-Q8_0.gguf`

**Lancement `llama-server`** :
```bash
llama-server --model ~/models/bge-m3/bge-m3-Q8_0.gguf \
  --embedding --pooling cls \
  --host 0.0.0.0 --port 8082 \
  --n-gpu-layers 99 --ctx-size 8192 --batch-size 512 --parallel 4
```

**Vérifié** : GB10 compute capability 12.1, 122 GB VRAM, modèle chargé 100% GPU, 4 slots parallèles.

## Découvertes importantes

1. **Le mode `think` de Qwen3** est LE piège majeur : sans `think: false`, le modèle consomme tous ses tokens de `num_predict` dans des blocs `<think>...</think>` invisibles dans la réponse. Solution API : `think: false` dans le body `/api/generate`.

2. **BGE-M3 sur GPU via llama.cpp gagne 200×** en conditions réelles (vs 40-60× prédit par le subagent). Le secret : le build llama.cpp avec `CMAKE_CUDA_ARCHITECTURES=121` active les kernels FP16 sur Grace Blackwell.

3. **Ollama gère déjà le LRU automatiquement** malgré `KEEP_ALIVE=-1`. Les modèles inactifs sont déchargés. Donc `MAX_LOADED_MODELS=2` suffit pour contrôler la contention.

4. **Le Classifier Käfer marche magnifiquement** : il reconnaît "ordinateur portable" → compte 1520 via le RAG Qdrant. Ça valide que l'ingestion du Käfer structuré est directement utile aux agents.

5. **Le benchmark officiel Ollama DGX Spark est 9.4 tok/s** pour Qwen3 32B Q4. On mesure **11.5 tok/s** sur Qwen3.5 27B Q4 → on est au-dessus grâce à `num_parallel=1` + `kv_cache q8_0`.

6. **Systemd > nohup** pour les services Spark : survie aux reboots, logs centralisés, restart auto, contrôle fin via `systemctl`.

7. **Le pipeline RAG complet fonctionne en 9.8s** pour une transaction simple. C'est **production-ready pour un MVP alpha**. Les 43s de `/rag/ask` peuvent être optimisées plus tard (modèle plus petit pour le reasoning).
