**Incident critique : EMBEDDER_URL cassé depuis session 12 (restauration silencieuse)**

Le `.env` prod sur `.59` pointait sur `EMBEDDER_URL=http://192.168.110.103:8001`,
mais ce port est un ancien uvicorn avec un endpoint `/embed` (non-OpenAI
compat). Le backend Lexa (`rag/EmbedderClient.ts`) appelle `/v1/embeddings`
depuis la session 11, format OpenAI. Le **bon** endpoint `/v1/embeddings`
est exposé par **llama-server** natif (service systemd `lexa-llama-embed.service`,
session 07) sur port **8082**.

Diagnostic via `ss -tlnp | grep -E "8001|8082"` sur `.103` :

```
LISTEN 0 2048 0.0.0.0:8001  uvicorn (ancien embedder, paths /embed + /health)
LISTEN 0 512  0.0.0.0:8082  llama-server (OpenAI compat, /v1/embeddings)
```

Le `/health` backend affichait `embedder: true` car la probe health est
superficielle (elle hit `/health` sur uvicorn qui répond `{ok:true}`).
Le bug était donc invisible jusqu'à ce qu'un `/rag/*` réel soit appelé.
**Session 12 a probablement restauré l'ancien .env par le rsync qui écrasait
`192.168.110.59:5432`** (autre bug session 12) et le EMBEDDER_URL n'a pas
été mis à jour en même temps.

**Fix** : `sed -i "s|EMBEDDER_URL=http://192.168.110.103:8001|EMBEDDER_URL=http://192.168.110.103:8082|" .env`
puis `pm2 restart lexa-backend`. RAG réparé instantanément, 5 citations
`AFC-INFO_TVA_12_TDFN` retournées.

**À retenir** : le health check backend doit valider un vrai call embedder
(`embedder.embedOne("test")`), pas juste un GET /health. À ajouter session 14
dans `routes/health.ts`.

### A.3 — Déclaration fiscale PP Valais (v1 PDF)

Deuxième formulaire de l'Execution layer après TVA, **premier à sortir du
domaine TVA** et à toucher le fiscal cantonal direct. C'est la sortie de
l'Execution mono-TVA vers le fiscal multi-domaines.

**Type : `VsPpFormTemplate` + `VsPpProjection` + `FilledVsPpForm`**

Structure **différente** de `FilledForm` (TVA) pour ne pas mélanger les
domaines. Champs :
- `revenuIndependant`, `revenuTotal`, `fortuneNette` (agrégats comptables)
- `fraisProForfait` (calculé : min(max(revenu × 3%, 2000), 4000))
- `deductionTotal`, `revenuImposable` (= total - déductions)
- `eventCount`

Pas de `period.kind`, car VS-PP est toujours annuel par construction — le
champ `year` est direct sur `FilledVsPpForm`.

**Template `01-knowledge-base/forms/vs-declaration-pp-2024.yaml`**

Fields organisés en sections (identité, période, revenus, fortune, déductions,
revenu imposable). La section `reference_amounts` contient les montants
officiels 2024 utilisés par le builder :

- `pilier_3a_salarie_max_chf: 7056`
- `pilier_3a_independant_max_chf: 35280`
- `frais_professionnels_forfait_pct: 3`, min 2000, max 4000

Les champs personnels (`revenu_salaire`, `pilier_3a`, `lpp_rachats`,
`primes_assurances`, `interets_dette`) sont marqués `todo: true` — ils
devront être complétés manuellement par l'utilisateur jusqu'à l'ajout
d'un onboarding personnel (session 14+).

**`VsPpFormBuilder.ts`**

Projection simplifiée depuis le grand livre Käfer :

```sql
-- Revenu net d'activité indépendante (bénéfice net)
revenuIndependant =
  Σ credit(3xxx) on year       -- produits
  − Σ debit(4-8xxx) on year    -- charges
```

```sql
-- Fortune nette au 31/12 (approximation)
fortuneNette =
  Σ (debit - credit) on accounts LIKE '1%'  -- actifs
  − Σ (credit - debit) on accounts LIKE '2%' -- passifs
```

Le frais pro forfait est calculé dans le builder via une règle dérivée du
template : `min(max(revenu × 0.03, 2000), 4000)`.

Mapping v1 : très pragmatique. Un comptable humain ne reconnaîtrait pas ces
chiffres comme une vraie déclaration fiscale, mais le flow est là, le
contribuable peut éditer le PDF à la main. Session 14+ : ajout d'un
onboarding personnel pour saisir revenu salarié, pilier 3a, etc.

**`VsPpPdfRenderer.ts`**

Nouveau renderer dédié (le TvaPdfRenderer est trop couplé à la structure
TvaProjection pour être réutilisé). Même style visuel (header, disclaimer
rouge, colonnes label/valeur, footer). 4 sections :

1. Contribuable (nom, UID, canton, année)
2. Revenus (indépendant + TODO salarié + TODO capital + total)
3. Fortune (nette au 31/12)
4. Déductions (frais pro forfait auto + 4 TODO : pilier 3a, LPP, assurances,
   intérêts) + total + revenu imposable

Les TODO affichent 0.00 CHF mais avec un label explicite "TODO : saisir
manuellement" pour signaler à l'utilisateur qu'il doit compléter.

**`idempotence.ts` étendu**

Deux nouvelles fonctions parallèles à celles TVA :

- `findExistingVsPpDeclaration({tenantId, formId, version, year})` — la clé
  d'idempotence VS-PP est **`(tenant, formId, version, year)`** seulement
  (pas de `method`, pas de `quarter`). Query SQL filtrée sur
  `payload->>'formKind' = 'vs-pp'`.
- `appendVsPpDeclarationEvent(form: FilledVsPpForm)` — construit le payload
  avec `formKind: 'vs-pp'`, `totals: {revenuTotal, fortuneNette,
  deductionTotal, revenuImposable}`.

**Type `LexaEvent.DeclarationGenerated` étendu**

Session 12 avait `totals` strictement typé avec les champs TVA. Session 13
le rend générique via `totals: Record<string, unknown>`, et ajoute deux
nouveaux champs :

- `formKind: 'tva' | 'vs-pp'` — permet aux queries d'idempotence de filtrer
  proprement sans risquer de collision entre domaines
- `method?: 'effective' | 'tdfn'` — maintenant optionnel (absent pour les
  formulaires non-TVA)

Le schéma JSONB reste permissif (tout est sous `payload`), donc les events
session 12 sans `formKind` continuent de fonctionner avec les queries
d'idempotence (qui filtrent sur `formId` + `version` + `method` + `year` +
`quarter`, pas sur `formKind` pour la compat rétroactive).

**Endpoint `POST /forms/vs-declaration-pp`**

Body zod : `{ year: number }`. Pas de `method`, pas de `sectorCode`. Les
paramètres personnels (statut, déductions) seront ajoutés session 14+ avec
l'onboarding personnel.

Pipeline :
1. `buildVsPpDeclaration({tenantId, year})`
2. `findExistingVsPpDeclaration(...)` → idempotence
3. `renderVsPpPdf(form)` (pas de XML en v1 — le canton VS n'exige pas
   d'échange structuré, `schema: none` dans le template)
4. `appendVsPpDeclarationEvent(...)` si pas d'existant
5. Retour `{ streamId, eventId, idempotent, form, pdf }`

**Frontend : bouton secondaire "Décl. PP VS" dans LedgerModal**

Deuxième bouton à côté du bouton TVA primaire, icône `Landmark` (lucide),
style `btn-secondary`. Click → appel `lexa.generateVsPpDeclaration({year})`
avec l'année courante du select (partagée avec TVA). Un seul blob PDF
téléchargé (pas de XML). Toast de succès avec le streamId et le suffixe

Les deux boutons (TVA et VS-PP) se désactivent mutuellement via
`disabled={generating || generatingVsPp}` pour éviter les doubles clicks.

## Décisions structurelles ratifiées

1. **Helpers execution mutualisés dans `shared.ts`** — décision effective :
   tout nouveau builder importe `getCompany`, `loadTemplate`, `projectTva` ou
   ses équivalents depuis `shared.ts`. Plus jamais de duplication. Les builders
   deviennent des wrappers minces de ~20 lignes.
2. **Idempotence par formKind** — chaque event `DeclarationGenerated` porte
   un champ `formKind: 'tva' | 'vs-pp'` qui permet aux queries d'idempotence
   de filtrer proprement sans risque de collision entre domaines. Extensible
   session 14+ avec `'vs-pm'`, `'ge-pp'`, etc.
3. **Un YAML par formulaire, un Builder par formulaire** — confirmé par la
   structure session 13 avec 4 templates YAML + 4 builders (Tva, TvaAnnual,
   Tdfn, VsPp). Pas de builder générique, pas de "mega-template" qui gère
   tous les cas. Chaque domaine fiscal reste isolé, plus facile à tester et
   à faire évoluer.
4. **Un Modelfile par canton pour les agents fiscaux** — confirmé par
   `lexa-fiscal-pp-vs` (session 13) et le plan `lexa-fiscal-pp-ge`
   (session 14+). L'orchestrateur routera selon le canton de l'activeCompany.
5. **qa-lexa comme baseline de régression** — le script devient la référence
   pour toutes les sessions futures. À chaque ajout de fonctionnalité, on
   ajoute une fixture. Pass rate < 100% = red flag à investiguer avant push.
6. **PDF pdfkit uniforme** — Tva et VsPp utilisent tous les deux pdfkit
   (décision #9 session 12 étendue). Pas d'introduction de @react-pdf/renderer
   ou autre.
