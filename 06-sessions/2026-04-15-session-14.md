**A.1 — Migration `004_users.sql`**

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email TEXT UNIQUE NOT NULL,
  password_hash TEXT NOT NULL,
  verified BOOLEAN NOT NULL DEFAULT false,
  tenant_id UUID NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  last_login_at TIMESTAMPTZ
);
```

Indexes sur `email` + `tenant_id`. Le champ `verified` est stocké mais
ignoré par `requireAuth` en v1 — permet de brancher Postmark ultérieurement
sans migration supplémentaire. Migration appliquée sur `.59` via
`tsx src/db/migrate.ts`.

**A.2 — `auth/jwt.ts` + bcrypt helpers**

Dépendances : `bcryptjs` (pure JS, évite la compilation native qui pose
problème avec `tsx watch` sur certains hosts), `jsonwebtoken`,
`express-rate-limit`. `@types/bcryptjs` et `@types/jsonwebtoken` en dev.

4 helpers exportés :
- `hashPassword(plain)` → bcrypt cost 12
- `comparePassword(plain, hash)` → bool
- `signToken(payload: {sub, tenantId, email})` → JWT HS256, `expiresIn: "7d"`
- `verifyToken(token)` → `JwtPayload` ou throw

Config étendu dans `config/index.ts` : `JWT_SECRET` (min 32 chars),
`JWT_EXPIRES_IN` (default `"7d"`), `ADMIN_SECRET` (min 16), `LEXA_WEBHOOK_SECRET`
(min 16 — préparation bloc C).

Secrets prod générés via `openssl rand -hex 32` et appendés au
`.env` de prod sur `.59`.

**A.3 — Routes `/auth/*`**

4 endpoints dans `routes/auth.ts` :

1. **`POST /auth/register`** — body `{email, password, company: {name, legalForm, canton, isVatSubject, ...}}`.
   Crée la company avec un `tenant_id` random UUID puis crée le user avec
   le même tenantId dans **deux INSERT** (pas de transaction explicite en v1,
   acceptable pour un produit à un digit d'users). Retourne
   `{user, company, token}`. Conflict 409 si email déjà enregistré.

2. **`POST /auth/login`** — body `{email, password}`, protégé par
   `express-rate-limit` 5 tentatives / 15 min / IP. Vérifie le hash,
   update `last_login_at`, retourne `{user, token}`. 401 sur credentials
   invalides.

3. **`GET /auth/me`** — protégé par `requireAuth`, retourne `{user, company}`
   avec jointure sur `companies` via `tenant_id`. Utilisé par le frontend
   pour rafraîchir le state sans re-login.

4. **`POST /auth/admin/reset-password`** — body `{email, newPassword}`,
   protégé par un header `X-Admin-Secret` comparé à `config.ADMIN_SECRET`.
   Filet de sécurité pour ne jamais bloquer un testeur sans password
   reset user-facing. 403 sans header, 200 + `{ok, userId}` sinon.

**A.4 — `middleware/requireAuth.ts`**

Lit `Authorization: Bearer <token>`, regex match strict, `verifyToken`,
inject `req.user = payload` et **override `req.tenantId = payload.tenantId`**
(le JWT a priorité sur l'ancien header `X-Tenant-Id`). 401 sur token absent
ou invalide. Module augmentation Express `Request.user?: JwtPayload`.

### Bloc C — Classifier auto Pro→Lexa + HMAC (~45 min)

**Important** : en étudiant le code existant, j'ai découvert que
**`classify` avait déjà le default `true`** dans le zod schema du
router `/connectors/bank/ingest` depuis la session 09. Les NEXT-SESSION
11/12/13 mentionnaient "classify:false" en se référant au *test manuel*
session 11, pas à l'état par défaut du hook. Donc la classification auto
était en fait techniquement déjà active dès que `LEXA_ENABLED=true` — mais
**invisible** parce que le cron IMAP `:30` ne tombe jamais pendant les
sessions actives (toujours obscurci par les restarts pm2). Et surtout,
**l'endpoint `/connectors/bank/ingest` n'avait aucune auth** : n'importe qui
sur le réseau LAN pouvait POST.

Session 14 corrige ça en ajoutant une **auth HMAC service-to-service** via
`X-Lexa-Signature` :

**`middleware/requireHmac.ts`**

- Regex match `/^sha256=([a-f0-9]{64})$/i` sur le header
- Calcule `HMAC-SHA256(LEXA_WEBHOOK_SECRET, rawBody)` côté backend
- Comparaison en **timing-safe** via `crypto.timingSafeEqual` pour éviter
  les attaques de timing
- 401 sur header absent/malformé ou signature non matching
- 500 si `req.rawBody` n'a pas été capturé (= hook `express.json({verify})`
  manquant — garde-fou)

**`app.ts`** : le body parser `express.json()` est enrichi avec un hook
`verify` qui capture le raw buffer dans `(req as any).rawBody` avant que
le stream ne soit consommé par le JSON parser. Nécessaire pour que le HMAC
soit calculé sur les bytes **exacts** reçus, pas sur une reserialization.

**`routes/connectors.ts`** — `POST /bank/ingest` devient
`post("/bank/ingest", requireHmac, async ...)`. Un seul middleware à
insérer, reste 100% compatible avec le payload existant.

**Patch `swigs-workflow/backend/src/services/bankImapFetcher.service.js`**

`pushTransactionToLexa(tx, userId)` réécrit :
- Ajoute `classify: true` explicite dans le payload
- `const rawBody = JSON.stringify(payload)` calculé AVANT l'appel
- `createHmac('sha256', LEXA_WEBHOOK_SECRET).update(rawBody).digest('hex')`
- `axios.post(url, rawBody, { headers: { 'X-Lexa-Signature': 'sha256=...' } })`
- **Important** : on passe `rawBody` (string) à axios, pas l'objet — sinon
  axios reserializerait le JSON et le HMAC ne matcherait plus. En envoyant
  une string, axios l'envoie telle quelle tant qu'on set `Content-Type: application/json`
- Guard : `if (!LEXA_WEBHOOK_SECRET) { console.warn; return null }` pour
  ne jamais crasher Pro si le secret manque

Ajout de `LEXA_WEBHOOK_SECRET` dans `/home/swigs/swigs-workflow/.env` avec
la même valeur que `/home/swigs/lexa-backend/.env`. `pm2 restart
swigs-workflow --update-env` OK, logs confirment
"Bank IMAP service initialized (hourly check at :30)".

## Incidents traités en cours de session

### 1. `certbot --nginx` refuse une config avec cert manquant

Premier essai cassé : la config nginx référençait `ssl_certificate
/etc/letsencrypt/live/lexa.swigs.online/fullchain.pem` qui n'existait pas
encore. `nginx -t` échouait, donc `certbot --nginx` ne pouvait pas
l'utiliser comme point de départ pour enrichir le vhost.

Fix : méthode `webroot` en 2 étapes (config HTTP-only → webroot → config
HTTPS finale). Plus prévisible que le plugin nginx et compatible avec les
vhosts existants (pas de risque de mutation accidentelle).

À noter pour les futurs deploys : **toujours commencer par une config
HTTP-only** qui sert `/.well-known/acme-challenge/`, faire `certbot
--webroot`, puis ajouter le bloc 443 avec SSL.

### 2. `classify: true` était déjà le default — pas de patch session 14

En lisant `routes/connectors.ts` j'ai vu que le zod schema avait
`classify: z.boolean().default(true)`. Le "patch classify:true" annoncé dans
le briefing session 14 était déjà en place côté backend depuis la session 09.
La sessions 11/12/13 parlaient de "classify:false" en référence au test
manuel session 11, pas au default.

**Ce qui manquait vraiment** : l'auth HMAC sur `/connectors/bank/ingest` (fait)
et l'ajout du header HMAC côté Pro (fait). Le vrai patch session 14 est
donc la sécurisation service-to-service, pas le flag `classify`.

## Décisions structurelles ratifiées

1. **bcryptjs vs bcrypt natif** : choix de `bcryptjs` (pure JS) pour éviter
   la compilation native qui peut casser avec `tsx watch` sur certains
   hosts. Cost 12 identique à bcrypt natif, latence légèrement plus élevée
   (~50ms vs ~20ms) mais négligeable sur un login.

2. **JWT dans `Authorization: Bearer` uniquement**, pas de cookies. Simple,
   évite la config CSRF, compatible avec `curl` et les agents de test.
   Session 16+ pour l'ajout optionnel de cookies `HttpOnly` pour certains
   parcours sensibles.

3. **Override `req.tenantId` par le JWT** — le middleware `requireAuth`
   prend le tenant du token, ignore le header `X-Tenant-Id` côté protégé.
   Le header reste utile pour les routes publiques (search UID register
   avant inscription) mais ne peut **plus** être utilisé pour usurper un
   autre tenant une fois authentifié.

4. **Endpoint admin reset avec header secret partagé** — pas de flow
   self-service password reset en v1 (ça demanderait email + token temp +
   page de reset). Le header admin suffit pour un testeur qui oublie son
   password, et permettra de brancher Postmark plus tard sans refonte.

5. **Rate limit uniquement sur `/auth/login`** — les autres endpoints
   protégés par JWT n'ont pas de risque de bruteforce (le token est une
   clé de 256 bits). Session 15+ pour `/auth/register` si un bot exploite
   l'inscription libre. Pour l'instant la cible est les bêta-testeurs
   humains, pas le public anonyme.

6. **HMAC service-to-service strictement séparé du JWT user-facing**.
   `/connectors/bank/ingest` n'a AUCUNE auth JWT, uniquement HMAC. C'est
   délibéré : le pont Pro n'a pas d'utilisateur humain, c'est un process
   cron. L'utiliser via JWT impliquerait de provisionner un "user Pro"
   avec des credentials qui traînent dans un process, ce qui est pire.

7. **Raw body capture via `verify` hook d'express.json** — pattern
   canonique pour HMAC sur du JSON. Alternative (`bodyParser.raw` puis
   parse manuel) est plus lourde et casse les handlers existants. Le hook
   `verify` est zero-invasion : les handlers voient toujours `req.body`
   parsé, le middleware HMAC voit `req.rawBody` buffer.

8. **Deploy frontend en deux étapes HTTP → HTTPS pour Let's Encrypt**
   quand le plugin certbot-nginx n'est pas déjà configuré. À documenter
   comme runbook pour les futurs vhosts.

9. **`lexa.swigs.online` = premier subdomain Lexa public**. Session 14 pose
   le pattern pour les futurs subdomains (ex: `api.lexa.swigs.online` si
   on sépare backend/frontend, `hub.lexa.swigs.online` pour le mode
   fiduciaire). Pour l'instant un seul host avec proxy `/api`.
