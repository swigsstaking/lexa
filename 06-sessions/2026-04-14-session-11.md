### 1. Dark mode architecture

**Plus de palette `lexa-*`** — remplacée par des **tokens sémantiques CSS variables**
(`--bg`, `--surface`, `--elevated`, `--border`, `--border-strong`, `--ink`,
`--muted`, `--subtle`, `--accent`, `--accent-fg`, `--success`, `--warning`,
`--danger`), déclinées en `.light` et `.dark` dans `index.css`. Tailwind config
mappe chaque token vers `bg-bg`, `text-ink`, etc. via la syntaxe alpha-value
(`rgb(var(--bg) / <alpha-value>)`).

`<html lang="fr" class="dark">` par défaut dans `index.html`. Fonts chargées via
Google Fonts : **Inter** (UI) + **JetBrains Mono** (données financières
`.mono-num` avec `tabular-nums`). Fraunces supprimée.

Composants utilitaires refaits : `.btn-primary`, `.btn-secondary`, `.btn-ghost`,
`.card`, `.card-elevated`, `.input`, `.label`, `.chip`, `.kbd`, `.mono-num`.
Scrollbars custom, selection bg-accent/25.

Screenshot : `06-sessions/assets/session-11-home-dark.png` — Home dark mode avec
hero "La compta PME suisse, pilotée par l'IA" en Inter, accent rouge.

### 10. Fix onboarding bug — Company sans tenantId

**Bug découvert au smoke test** : après onboarding, l'URL restait sur
`/onboarding` au lieu de naviguer vers `/workspace`. Cause : le backend POST
`/onboarding/company` retournait `{ id, tenantId, createdAt, company: data }`
mais le frontend lisait `.data.company` qui était juste le payload d'input,
**sans tenantId ni id**. Donc `addCompany(companyWithoutTenantId)` ne pouvait
pas set `activeCompanyId`, `useActiveCompany()` retournait null, et
`RequireCompany` redirigeait vers `/onboarding`.

**Fix backend** : nouveau format de réponse unifié `{ company: { id, tenantId,
...data, createdAt, updatedAt } }`. Frontend inchangé, il reçoit maintenant un
Company complet.

Après fix + clear localStorage + retry du flow : Kozelsky Sàrl → page
`/workspace` avec canvas, header Kozelsky/VS.

## Décisions structurelles ratifiées

1. **Tokens sémantiques CSS variables > palette nominale** — on peut theme-swap
   light/dark/autres en changeant juste les variables, zero refactor
   composant. Si demain on veut un "mode éco" avec contraste max pour malvoyants,
   c'est 10 lignes de CSS.

2. **Multi-tenant dès le départ** — le store est pluriel, le middleware backend
   accepte le header, l'axios interceptor propage automatiquement. Pas de
   "temporary single-tenant" à refacto plus tard. Résout P3 session 10 en même
   temps.

3. **Tous les strings passent par i18next** — même en FR seul v1, on gagne
   énormément à poser le pipeline maintenant. Ajouter DE = créer `de.json`, un
   point.

4. **Workspace = UNE route** — pas de sidebar, pas de menu, pas de tabs. Le
   canvas EST l'interface. Le chat EST l'assistant ubiquitaire. Le grand livre
   classique EST un fallback expert. C'est la promesse whitepaper littéralement.

5. **react-flow choisi sans benchmark** — sondage technique sur la doc v12 + la
   ref whitepaper "nœuds vivants, arêtes animées, clusters" = mapping direct.
   Custom nodes + edges = domain-modeling propre. tldraw aurait forcé à
   reconstruire tout le layer nodes/edges à la main.

6. **ChatOverlay portal, pas route** — ça permet au chat de rester actif pendant
   qu'on regarde le canvas, de dialoguer avec lui sans quitter le contexte
   visuel. C'est le cœur de "conversationnel first".

7. **Ledger = toggle, pas route** — celui qui veut la vue tableau classique
   presse `cmd+shift+L`. Par défaut, le canvas. Whitepaper §5 : "Modes de vue
   togglable (Canvas défaut, …, Livres fallback)".

8. **Pas de dépendances UI lourdes** — on n'a PAS installé shadcn/ui,
   radix-ui, headless-ui, ou material-ui. Tout est custom Tailwind + lucide
   icons + framer-motion. Ça reste < 700 KB JS gzippé 225 KB. Si shadcn devient
   utile en session 12+, on l'ajoutera ciblément.
