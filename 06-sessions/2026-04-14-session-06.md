### 2. Event store Postgres

**Migration 001_events.sql** :

```sql
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

CREATE TABLE events (
  id BIGSERIAL PRIMARY KEY,
  tenant_id UUID NOT NULL,
  stream_id UUID NOT NULL,
  sequence BIGINT NOT NULL,
  type TEXT NOT NULL,
  payload JSONB NOT NULL,
  metadata JSONB NOT NULL DEFAULT '{}'::jsonb,
  occurred_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  recorded_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  CONSTRAINT events_stream_seq_unique UNIQUE (tenant_id, stream_id, sequence)
);
CREATE INDEX idx_events_tenant_stream ...;
CREATE INDEX idx_events_type ...;
CREATE INDEX idx_events_payload_gin ON events USING GIN (payload);

CREATE TABLE ai_decisions (
  id UUID PRIMARY KEY,
  event_id BIGINT REFERENCES events(id),
  tenant_id UUID, agent TEXT, model TEXT,
  confidence NUMERIC(5,4), reasoning TEXT,
  citations JSONB, alternatives JSONB, rag_context JSONB,
  duration_ms INTEGER, created_at TIMESTAMPTZ
);

CREATE TABLE schema_migrations (version TEXT PRIMARY KEY, applied_at TIMESTAMPTZ);
```

## Décisions structurelles prises

1. **Mono-repo** : `lexa/apps/backend/` (plus tard : `lexa/apps/frontend/`, `lexa/apps/worker/`, etc.). Simpler qu'un multi-repo pour l'instant.

2. **Stack backend** : Node 20 + TypeScript strict (ES2022, Bundler resolution) + Express 4 + Zod + pg + axios + pino + tsx. Pas de framework lourd (NestJS, Fastify) — on reste minimal.

3. **Event store = Postgres + JSONB** : pas d'outil dédié (EventStoreDB, Kafka). `events` table avec unique constraint `(tenant_id, stream_id, sequence)`. GIN index sur payload pour search.

4. **Micro-service BGE-M3 en Python** : inévitable car FlagEmbedding n'existe qu'en Python. FastAPI + uvicorn, 1 worker, port 8001, exposé en HTTP sur le Spark. Le backend TS appelle via HTTP.

5. **Modèle pour agents** : par défaut `qwen3.5:27b-q8_0` (base, sans system prompt lourd) pour le Classifier. `comptable-suisse` (fine-tuné) pour les réponses conversationnelles où la prose + citations est OK. Un `lexa-classifier` Modelfile dédié sera créé session 07.

6. **Timeouts Ollama 900s** : le Spark est lent sous contention (les 3 modèles chargés en parallèle se partagent le GPU). Pas un bug, un throttling.

7. **Secrets hors git** : `.env` dans `.gitignore`, password Postgres dans `~/.lexa_db_pass_temp` sur Mac + `/home/swigs/lexa-backend/.env` (mode 600) sur .59.

## Découvertes importantes

1. **Le Spark GB10 est lent sous contention GPU**. Une simple requête Ollama `comptable-suisse-fast` (Q4) prend **4m18s pour ~200 tokens = 0.8 tok/s** au lieu des 11 tok/s mesurés en session 01. Cause probable : les 3 modèles chargés en mémoire (comptable-suisse-fast + qwen3.5:9b + deepseek-ocr) se battent pour le compute GPU.

2. **Ollama `format: "json"` bug avec Qwen3 + thinking mode** : le modèle retourne une réponse vide. Workaround : parser la prose naturelle avec regex tolérants, ou créer un Modelfile custom sans thinking mode.

3. **Le SYSTEM prompt de `comptable-suisse-fast` est énorme** : il contient le plan Käfer complet (~25k tokens). Ça bouffe tout le contexte et fait échouer les prompts custom longs. Il faut un modèle dédié à la classification sans ce SYSTEM.

4. **Le pipeline BGE-M3 via HTTP fonctionne parfaitement**. Le micro-service FastAPI (~50 lignes) charge le modèle en 30s au démarrage, puis répond en ~500ms par batch de 16 embeddings. Pattern réutilisable pour exposer d'autres modèles Python au backend TS.

5. **Postgres event store + JSONB est suffisant** pour démarrer. Pas besoin d'EventStoreDB ou Kafka. Les index GIN sur payload permettent des queries JSONPath efficaces pour les projections.

6. **Mono-repo > multi-repo** pour Lexa à ce stade. Un seul `git push` synchronise docs + code + scripts. Le jour où le frontend arrive, il ira aussi dans `apps/frontend/`.
