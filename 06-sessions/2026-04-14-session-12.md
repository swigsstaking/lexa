### B. Execution layer — décompte TVA AFC trimestriel

Le gros morceau de la session. 6 sous-tâches B1-B6.

**B1 — Template YAML déclaratif**

`01-knowledge-base/forms/tva-afc-decompte-effectif-2024.yaml`. Structure
conforme whitepaper §3 couche 4 :

- `form_id`, `version`, `title`, `authority: AFC`, `legal_reference` (LTVA
  RS 641.20, articles 36/37/71)
- `vat_rates` : trois taux 2024 (standard 8.1%, réduit 2.6%, hébergement 3.8%)
- `fields` : 18 champs déclaratifs avec `source: projection.tva.X` ou
  `company.X` / `input.X` / `computed.X`, + `validation` required/optional, +
  `format: currency_chf`, + `todo: true` pour les champs v2 (investissements,
  exonéré…)
- `output` : `pdf.renderer: pdfkit` + `pdf.disclaimer` (texte complet du
  marquage Lexa) + `xml.schema: eCH-0217`
- `validation.human_signature_required: true` +
  `validation.lexa_liability: preparation_only`

Le fichier est source de vérité dans `01-knowledge-base/forms/`. Une **copie
embed runtime** a été placée dans `apps/backend/src/execution/templates/`
pour éviter les paths cross-repo au runtime tsx/ESM (voir B2).

**B2 — Service `TvaFormBuilder.ts`**

`apps/backend/src/execution/TvaFormBuilder.ts` :

- Charge le template YAML avec `yaml` npm package (`parse`), cache en mémoire
  après le premier load
- `getCompany(tenantId)` : query `companies WHERE tenant_id`. Si le tenant n'a
  pas de row (tenants historiques créés avant la migration 003), retourne une
  `CompanyInfo` minimale `{ name: "Tenant XXXXXXXX", ... null }` — le PDF et
  XML afficheront "—" dans les champs manquants, le flow reste utilisable
- `projectTva(tenantId, start, end)` : SQL sur la materialized view
  `ledger_entries` (déjà utilisée par `/ledger/balance`), filtrage par
  `transaction_date BETWEEN $2 AND $3`
- Logique de projection :
  - Ligne `line_type='credit'` sur compte produit (`account LIKE '3%'`) →
    vente : `projection.caHt[code] += amountHt`,
    `projection.tvaDue[code] += amountTtc - amountHt`
  - Ligne `line_type='debit'` sur compte charge (`4`/`5`/`6`/`7`/`8`) avec
    `rate > 0` → impôt préalable opérationnel
  - Rate code determiné par `rateCodeFor()` (0.05 de tolérance sur 8.1/2.6/3.8)
- `buildDecompteTva({ tenantId, quarter, year, method })` assemble le
  `FilledForm` avec company, period, projection, template, generatedAt
- `quarterRange(q, y)` calcule les dates début/fin du trimestre civil

**B3 — Endpoint `POST /forms/tva-decompte`**

`apps/backend/src/routes/forms.ts`, monté dans `app.ts` avec `app.use("/forms", formsRouter)`.

- Body zod schema : `{ quarter: 1|2|3|4, year: 2020-2100, method?: "effective"|"tdfn" }`
- Lit `tenantId` depuis `req.tenantId` (middleware session 11 déjà en place)
- Pipeline : `buildDecompteTva` → `renderDecompteTvaPdf` + `renderDecompteTvaXml`
- **Audit trail** : event `DeclarationGenerated` inséré dans l'event store
  avec `eventStore.append`, streamId random UUID, payload contenant totals +
  liability `preparation_only`
- Retour : `{ streamId, eventId, form, pdf: base64, xml: string }`
- Error handler propre : 500 + message explicite

Type `LexaEvent` étendu dans `events/types.ts` avec `DeclarationGenerated`
(payload typé strictement : formId, period, totals, eventCount, liability).

**B4 — PDF pdfkit avec disclaimer obligatoire**

`apps/backend/src/execution/TvaPdfRenderer.ts`. Choix **pdfkit** (pas
`@react-pdf/renderer`) confirmé par la décision tranchée #9.

Template A4 portrait, marges 48px, Helvetica, sections :

1. Header : titre "Décompte TVA — méthode effective" + sous-titre AFC +
   référence LTVA articles
2. **Disclaimer box** rouge (`#fef2f2` fond, `#ef4444` border, texte `#991b1b`) :
   texte exact du YAML `output.pdf.disclaimer` — marquage OBLIGATOIRE
   whitepaper §6 phase 1
3. Entreprise : raison sociale, UID, numéro TVA, canton
4. Période : trimestre, dates début/fin, méthode
5. Table "Chiffre d'affaires imposable" : 3 lignes (8.1% / 2.6% / 3.8%) avec
   colonnes Description / CA HT / TVA due (alignement `width+align: right`)
6. Total TVA due (Helvetica-Bold)
7. Impôt préalable (opérationnel + total)
8. Solde à payer (label dynamique "à payer" ou "à recevoir" selon signe)
9. Footer : timestamp généré + formId + v + nombre d'écritures agrégées

Export en Buffer via `doc.on('data' | 'end')` et Promise.

**B5 — XML eCH-0217 minimal v1**

`apps/backend/src/execution/TvaXmlBuilder.ts`. Pas de libxml : string builder
avec `escapeXml()`. Structure :

```xml
<VATDeclaration xmlns="http://www.ech.ch/xmlns/eCH-0217/1"
                version="1.0"
                status="draft"
                generatedBy="Lexa"
                generatedAt="...">
  <enterprise>…</enterprise>
  <period>…</period>
  <taxableTurnover>
    <rate code="standard" percent="8.1">
      <netAmount>…</netAmount>
      <vatDue>…</vatDue>
    </rate>
    … (reduced, lodging)
    <totalVatDue>…</totalVatDue>
  </taxableTurnover>
  <inputTax>
    <operatingExpenses>…</operatingExpenses>
    <capitalExpenditure>…</capitalExpenditure>
    <total>…</total>
  </inputTax>
  <balance>
    <amountDue>…</amountDue>
    <direction>toAuthority|fromAuthority</direction>
  </balance>
  <disclaimer>
    <liability>preparation_only</liability>
    <note>…</note>
  </disclaimer>
</VATDeclaration>
```

Marquage `status="draft"` explicite sur la racine pour éviter toute confusion
avec un dépôt officiel. Commentaire TODO dans le code listant les blocs
manquants pour v2 (acquisitions, reductions, selfSupply, exports, tax-exempt
turnover, corrections, prior period adjustments).

## Incidents traités en cours de session

**Incident .env écrasé par rsync --delete**

Le premier `rsync -avz --delete` depuis `apps/backend/` vers `.59:/home/swigs/lexa-backend/`
a copié le `.env` local qui pointait sur `192.168.110.59:5432`, alors que
PostgreSQL sur .59 n'écoute que sur `127.0.0.1:5432` (`ss -tlnp | grep 5432`
→ `LISTEN 127.0.0.1:5432`). Résultat : backend en crash loop `ECONNREFUSED
192.168.110.59:5432`.

**Fix** : `sed -i "s|192.168.110.59:5432|127.0.0.1:5432|g"` sur le `.env` prod
puis `pm2 restart lexa-backend --update-env`. Services verts dans les 3s.

**Leçon** : toujours exclure `.env` des rsync de déploiement, ou rsync juste
le sous-dossier `src/`. Les commandes de déploiement ultérieures ont utilisé
`rsync --exclude=.env --exclude=node_modules` et seulement `src/` en source.

**Incident tenant seed sans company row**

Premier POST `/forms/tva-decompte` → 500 "No company found for tenant
00000000-…". Le tenant seed a été créé avant la migration 003_companies
(session 09), donc il a des events `TransactionClassified` mais pas de row
`companies`.

**Fix** : `TvaFormBuilder.getCompany()` émet maintenant une `CompanyInfo`
minimale quand le SELECT retourne 0 row, au lieu de throw. Le builder reste
utilisable, le PDF/XML affichent `Tenant XXXXXXXX` et "—" dans les champs
absents. Comportement cohérent avec les tenants historiques.

## Décisions structurelles ratifiées

1. **Template = source canonique + copie runtime embed**. Le YAML vit dans
   `01-knowledge-base/forms/` (whitepaper §3 couche 4), mais le backend charge
   depuis `src/execution/templates/` pour éviter les chemins cross-repo
   `../../../../` au runtime tsx. Synchronisation manuelle en attendant un
   step de build qui copiera automatiquement.
2. **Fallback company pour tenants orphelins**. Plutôt que de bloquer avec
   un 500, le builder émet une `CompanyInfo` minimale. Plus résilient, et
   cohérent avec le fait qu'un tenant peut exister uniquement via header
   `X-Tenant-Id` en v1.
3. **Event `DeclarationGenerated`** : ajouté au `LexaEvent` avec payload typé
   strict. Chaque génération de décompte devient un fait immuable dans
   l'event store, interrogeable via `eventStore.readByType("DeclarationGenerated")`
   pour l'audit trail et la reprise (idempotence TBD en session 13).
4. **Marquage PDF/XML "draft" + disclaimer obligatoire**. Whitepaper §6 phase
   1 : Lexa *prépare*, l'humain *valide et signe*. Texte exact dans
   `template.output.pdf.disclaimer`, rendu en box rouge dans le PDF et
   comme `<disclaimer><liability>preparation_only</liability></disclaimer>`
   dans le XML. Non retirable sans édition explicite du template YAML.
5. **Toasts inline dans LedgerModal**, pas de librairie externe. Cohérent
   avec la décision "pas de dépendances UI lourdes" (session 11). Banner
   `bg-success/10` ou `bg-danger/10` dans le header du modal, auto-dismiss
   5s.
