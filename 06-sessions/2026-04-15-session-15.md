## Travail réalisé

### A.1 — Migration `005_taxpayer_drafts`

Table unique keyée sur `(tenant_id, fiscal_year)` avec état JSONB flexible :

```sql
CREATE TABLE taxpayer_drafts (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id     UUID NOT NULL,
  fiscal_year   INTEGER NOT NULL,
  state         JSONB NOT NULL DEFAULT '{}'::jsonb,
  current_step  INTEGER NOT NULL DEFAULT 1,
  completed_at  TIMESTAMPTZ,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE (tenant_id, fiscal_year)
);
```

Index sur `tenant_id`, trigger `updated_at` automatique. Un draft par tenant
par année fiscale (contrainte UNIQUE composite). Le JSONB permet au schéma
d'évoluer (session 16+ GE-PP) sans nouvelle migration.

Appliquée sur `.59` : `schema_migrations` passe à 5.

### A.2 — Event `TaxpayerFieldUpdated` + schéma Zod

Nouveau type d'event dans `events/types.ts` :

```typescript
| {
    type: "TaxpayerFieldUpdated";
    payload: {
      fiscalYear: number;
      step: number;
      field: string;       // dot-path (ex: "step1.firstName")
      value: unknown;
      updatedBy: string;   // user.sub du JWT
    };
  }
```

Chaque mutation champ émet un event pour l'audit trail (whitepaper §2
event-sourcing). Best-effort : si l'append échoue, la mutation persistée
en table ne casse pas.

Nouveau fichier `taxpayers/schema.ts` avec 4 schémas Zod partiels :

- **Step1Identity** : firstName, lastName, dateOfBirth, civilStatus (6 valeurs),
  childrenCount (0-20), commune (string), canton (literal "VS")
- **Step2Revenues** : isSalarie, salaireBrut, hasSwissdecCertificate,
  revenusAccessoires, rentesAvs/Lpp/3ePilier, allocations, revenusTitres,
  revenusImmobiliers
- **Step3Wealth** : comptesBancaires, titresCotes, titresNonCotes,
  immeublesValeurFiscale, immeublesEmprunt, vehicules, autresBiens, dettes
- **Step4Deductions** : pilier3a, primesAssurance, fraisProFormat
  ("forfait" | "reel"), fraisProReels, interetsPassifs, rachatsLpp,
  fraisMedicaux, dons

Tous les champs sont **optionnels** dans le schéma partiel (wizard en cours),
validés au moment du PATCH via `TaxpayerDraftStateSchema.parse()` qui
catch les types invalides.

### A.3 — Service + routes `/taxpayers/draft`

**`taxpayers/service.ts`** expose 4 fonctions pures :

- `getOrCreateDraft(tenantId, fiscalYear)` — SELECT puis INSERT vide si
  inexistant
- `updateField({tenantId, userId, fiscalYear, step, field, value})` —
  applique une mutation dot-path (split `field` sur `.`, descend dans
  l'objet, set la feuille), valide le state complet via Zod, UPDATE,
  append event `TaxpayerFieldUpdated` (best-effort)
- `markSubmitted(tenantId, fiscalYear)` — set `completed_at = NOW()`
- `resetDraft(tenantId, fiscalYear)` — DELETE pour les tests/reprise

**`routes/taxpayers.ts`** avec 4 endpoints protégés par `requireAuth` :

1. **`GET /taxpayers/draft?year=2026`** — retourne ou crée le draft, avec
   `req.tenantId` depuis le JWT (isolation stricte)
2. **`PATCH /taxpayers/draft/field`** — body validé par
   `TaxpayerFieldUpdateSchema`, inject `userId = req.user.sub`
3. **`POST /taxpayers/draft/submit`** — pipeline : fetch draft →
   `buildVsPpDeclaration({tenantId, year, draft: state})` →
   `findExistingVsPpDeclaration` (idempotence) → `renderVsPpPdf` →
   `appendVsPpDeclarationEvent` si nouveau → `markSubmitted` → retour
   `{streamId, eventId, idempotent, form, pdf}`
4. **`POST /taxpayers/draft/reset`** — body `{fiscalYear}`, DELETE

Mount dans `app.ts` : `app.use("/taxpayers", requireAuth, taxpayersRouter)`.

### A.4 — Adapter `VsPpFormBuilder` au `TaxpayerDraft`

Le builder session 13 avait une projection hardcoded sur le grand livre
(revenu indépendant = produits − charges, fortune = actifs − passifs).
Session 15 introduit un **dual mode** :

- Mode `draft` : les valeurs viennent du wizard (step2-4 remplis). Chaque
  champ est pris directement : `revenuSalaire = step2.salaireBrut`,
  `fortuneBrute = Σ step3`, `deductionPilier3a = step4.pilier3a`, etc.
- Mode `ledger` (legacy) : si aucun step du draft n'est rempli, fallback
  sur la projection ledger session 13
- Mode `mixed` : si le draft a un revenu accessoire ET le ledger a des
  écritures non nulles, les deux sources coexistent

Le type `VsPpProjection` est étendu pour exposer **toutes les lignes** :
revenuSalaire, revenuIndependant, revenuAccessoires, revenuRentes,
revenuCapital, revenuImmobilier, fortuneBrute/Dettes/Nette, 7 déductions,
et un champ `source: "draft" | "ledger" | "mixed"` pour l'audit.

Le frais pro forfait devient dynamique :

```typescript
function computeFraisPro(refAmounts, revenuBrut, format, fraisReels) {
  if (format === "reel" && fraisReels !== undefined) return round2(fraisReels);
  const base = Math.max(revenuBrut, 0);
  const rawForfait = base * (refAmounts.frais_professionnels_forfait_pct / 100);
  return round2(Math.min(Math.max(rawForfait, min), max));
}
```

**PDF renderer** : la section Revenus passe de 3 lignes (dont 2 TODO) à 5
lignes factuelles (salaire, indépendant, rentes, capital, immobilier).
La section Fortune affiche brute/dettes/nette au lieu du seul `fortuneNette`
brut. La section Déductions liste les 7 champs réels au lieu de 4 TODO.
Le footer affiche `source : saisie contribuable` au lieu de `X lignes
ledger agrégées` quand le draft est rempli.

### A.5 — Frontend wizard 6 steps

**Architecture** :

- `stores/taxpayerDraftStore.ts` — zustand avec `draft`, `currentStep`,
  `pendingPatches: Map<field, timer>`, et méthodes `fetch`, `setStep`,
  `updateField` (optimistic + debounce 500ms), `reset`
- `routes/taxpayer/TaxpayerWizard.tsx` — shell du wizard avec header
  company info + stepper 6 boutons cliquables + main flex avec col-span-2
  (step actif) + col-span-1 (side-panel WizardSummary sticky top-6) +
  dialog exit confirm
- `routes/taxpayer/WizardSummary.tsx` — calcul live des totaux
  (revenuTotal, fortuneNette, deductionTotal, revenuImposable), pas de
  round-trip backend. Même sectionnement que le PDF pour que le user
  comprenne ce qui sortira.
- 6 composants steps dans `routes/taxpayer/steps/` :
  - **Step1Identity** : grid 2-col avec prénom/nom, date de naissance,
    état civil select, enfants à charge number, commune VS select (25
    communes hardcoded dans `data/communes-vs.ts`)
  - **Step2Revenues** : toggle salarié/indépendant avec sous-option
    "certificat Swissdec disponible", puis 7 `CurrencyField` (salaire,
    accessoires, 3 rentes, capital, immobiliers)
  - **Step3Wealth** : 8 `CurrencyField` (comptes, titres cotés/non cotés,
    immeubles valeur+emprunt, véhicules, autres, dettes)
  - **Step4Deductions** : pilier 3a avec plafond dynamique selon
    `isSalarie` (7056 vs 35280), rachats LPP, primes, radio
    forfait/réel pour frais pro, intérêts, médicaux, dons
  - **Step5Preview** : 5 checks d'éligibilité cliquables (identité,
    état civil, au moins un revenu, fortune, frais pro format), badge
    vert/warning + bouton retour au step concerné
  - **Step6Generate** : gros CTA "Générer ma déclaration PDF", appelle
    `lexa.submitTaxpayerDraft`, download auto du blob, affiche le résultat
    (revenuImposable + fortuneNette en grand), avec disclaimer rappel
    "Lexa prépare, humain signe"
- `CurrencyField.tsx` — component réutilisable input[type=number] avec
  suffixe `CHF` en absolute positioning et support optionnel d'un `max`

**Routing** : nouvelle route `/taxpayer/:year` dans `App.tsx`, wrap
`RequireAuth`. Accessible depuis le header Workspace via un nouveau
bouton `FileSignature` "Déclaration PP" qui navigate vers
`/taxpayer/${currentYear}`.

**Store debounce pattern** : chaque `updateField` applique une mutation
optimiste locale immédiate (pas de lag visuel), puis debounce 500ms avant
de PATCH le backend. Si l'user tape vite, les PATCH précédents sont
clearTimeout. Le `Map<field, timer>` garantit qu'on n'a pas de race
condition entre deux champs différents (chacun a son timer).

**Compile** : `tsc -b` clean. Build Vite : 743 KB JS / 234 KB gzip (+27
KB vs session 14 à cause du wizard). Le warning 500 KB persiste, dette
code-splitting reportée session 17+.

- **Détection d'un bug warning** : `express-rate-limit ValidationError
  ERR_ERL_UNEXPECTED_X_FORWARDED_FOR`. Derrière nginx, Express voit le
  header `X-Forwarded-For` mais `trust proxy` est à `false`, donc le
  rate limiter refuse de l'utiliser comme clé et warn en continu.

**Impact production** : sans `trust proxy`, le rate limiter compte
**toutes les requêtes comme venant d'une seule IP** (l'IP loopback de

le 429 global pour tous les utilisateurs. Bug de sécurité critique qui
aurait été invisible jusqu'à ce qu'un premier testeur se retrouve
bloqué en prod sans raison.

**Fix** (bloc D dérive mais urgent) :

```typescript
app.set("trust proxy", 1);
```

Ajouté au début d'`app.ts`, avant le body parser. Le `1` veut dire "un
hop proxy de confiance" (nginx sur le même host). Plus sûr que `true`
qui ferait confiance à tous les proxies.

**Deuxième observation (06:49 → push)** : relancée après le fix, tourne
jusqu'à la fin de session. Au moment du push, `:30` le plus récent est
à 07:30 — l'observation courra jusqu'à au moins un tick. Analyse au
moment du kill juste avant commits (voir section « Résultats observation

## Décisions structurelles ratifiées

1. **Un draft par tenant par année fiscale** (contrainte UNIQUE). Évite la
   complexité des drafts multiples en v1. Session 17+ pour les cas
   spéciaux (amendement rétroactif).
2. **State JSONB flexible** — même table pour VS-PP, GE-PP (session 16),
   VD-PP, etc. Le schéma évolue sans migration. Coût : les queries
   SQL sur le contenu du state sont plus lourdes, mais en v1 on ne
   query que par tenant+year.
3. **Mutation atomique par dot-path** — `PATCH /draft/field` prend
   `field: "step1.firstName"` et descend dans l'objet. Plus simple qu'un
   PATCH multi-fields et évite les race conditions entre champs.
4. **Event `TaxpayerFieldUpdated` best-effort** — si l'append échoue, la
   mutation en table ne casse pas (try/catch + console.warn). Le wizard
   reste utilisable même si l'event store a un souci temporaire. Audit
   trail = best-effort en v1, session 17+ pour le rendre strict avec
   un retry queue.
5. **Side-panel preview calculé côté client** — pas de round-trip
   backend à chaque frappe. Le calcul client réplique la logique
   `computeFraisPro` et les totaux. Le **vrai** PDF vient du backend à
   l'étape 6 avec les vraies règles fiscales et le disclaimer.
6. **Dual mode `draft | ledger | mixed`** dans `VsPpFormBuilder` avec le
   flag `source` affiché dans le PDF footer. Le legacy ledger reste pour
   la démo indépendant sans wizard (tenant seed, dev, tests).
7. **`trust proxy 1`** — un seul hop (nginx). Pas `true` qui ferait
   confiance à tous les proxies. Session 16+ si on ajoute un CDN devant,
   augmenter à `2`.
8. **Compile-time TaxpayerDraft typé côté frontend** — recopié dans
   `api/lexa.ts` plutôt qu'importé via un shared-types package. Session 18+
   pour factoriser si nécessaire. Pour un duo backend/frontend dans un
   même monorepo, la duplication de types est acceptable.
