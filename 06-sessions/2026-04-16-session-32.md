## Bloc A — Migration + Backend core (~90 min)

### A.1 — Migration 008 `fiduciary_memberships`

Fichier : `apps/backend/src/db/migrations/008_fiduciary_memberships.sql`

- Table N:M `user_id ↔ tenant_id` avec rôle `owner | fiduciary | viewer`
- 2 index : `fm_user` + `fm_tenant`
- Backfill : 39 users existants → owners de leur tenant (INSERT…ON CONFLICT DO NOTHING)
- Exécutée en prod : `INSERT 0 39`

**Bonus** : `ALTER TABLE users ALTER COLUMN tenant_id DROP NOT NULL` pour permettre les users fiduciaires sans tenant propre.

### A.2 — MembershipService.ts

Fichier : `apps/backend/src/services/MembershipService.ts`

- `listUserMemberships(userId)` : LEFT JOIN companies pour `tenantName`
- `validateMembership(userId, tenantId)` : boolean
- `grantMembership(userId, tenantId, role)` : ON CONFLICT DO UPDATE
- `revokeMembership(userId, tenantId)` : DELETE

### A.3 — JWT étendu

Fichier : `apps/backend/src/auth/jwt.ts`

Type JwtPayload étendu :
```ts
{ sub, tenantId, activeTenantId?, memberships?, email }
```

Route `/auth/login` émet maintenant `activeTenantId + memberships[]` (snapshot au login).

### A.4 — requireAuth mis à jour

Fichier : `apps/backend/src/middleware/requireAuth.ts`

`req.tenantId = payload.activeTenantId ?? payload.tenantId` — priorité S32, fallback legacy.

### A.5 — Route POST /auth/switch-tenant

Valide membership via `validateMembership`, recharge les memberships, émet nouveau JWT avec `activeTenantId` mis à jour. Retourne `{ token, activeTenantId }`.

### A.6 — Route GET /fiduciary/clients

Retourne `{ clients: Membership[] }` avec tenantId, role, tenantName, addedAt.

### A.7 — Route POST /fiduciary/invite

Owner-only : vérifie rôle owner sur tenantId, cherche user par email, grant membership.

### Smoke tests confirmés

- `GET /fiduciary/clients` → `{ clients: [{ tenantId: demo, role: owner, tenantName: "Demo Sàrl" }] }`
- Login fiduciaire → JWT avec `memberships.length === 2`
- `GET /fiduciary/clients` fiduciaire → 2 clients (Demo Sàrl + Acme SA)
- `POST /auth/switch-tenant {tenantId: demo}` → nouveau JWT + `activeTenantId: demo`
- `GET /ledger/balance-sheet/2026` avec nouveau token → assets 228 975 CHF ✓

**Décision** : RLS préparée mais inactive en V1. Raison : migrer toutes les queries des 4 tables critiques vers `queryAsTenant()` avant d'activer RLS pour éviter de casser l'app.

### B.1 — Migration 009 (NO-OP documentée)

Fichier : `apps/backend/src/db/migrations/009_rls_prepared.sql`

SQL de référence préparé pour activation future. Le fichier exécute un `SELECT 'status'` inoffensif et documente les 4 tables à activer + la procédure.

### B.2 — queryAsTenant wrapper

Fichier : `apps/backend/src/db/postgres.ts`

Wrapper `queryAsTenant<T>(tenantId, sql, params)` : BEGIN + SET LOCAL app.active_tenant + QUERY + COMMIT + ROLLBACK on error. Non utilisé en V1 mais exporté et prêt.

**Dette S33+ pré-launch** : activer RLS sur `taxpayer_drafts, company_drafts, events, ai_decisions` après migration complète queryAsTenant.
