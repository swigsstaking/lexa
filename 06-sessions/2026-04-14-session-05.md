### 5. Subagent Info TVA — stratégie technique validée

Un 2ème subagent a exploré les 4 approches pour récupérer les Info TVA du portail JSF `gate.estv.admin.ch` et **a trouvé une solution simple et élégante** :

**Le portail expose un endpoint `pdfSelectionList.xhtml`** qui génère un PDF FR complet via POST — pas besoin de Playwright, pas besoin de cliquer 100 sections !

**Flow en 4 étapes** :
1. `GET tableOfContent.xhtml?publicationId=X` → cookies + `ViewState1`
2. `POST` même URL avec `j_idt20:j_idt24` (lien Français) → langue session = fr
3. `GET pdfSelectionList.xhtml?publicationId=X&lang=fr` → `ViewState2` + N checkboxes
4. `POST` avec toutes les checkboxes cochées → retourne directement `application/pdf` (FR)

**Instructions détaillées du subagent** : flow exact, pièges à éviter (ViewState dynamique, IDs de formulaire qui peuvent changer), fallback pour les IDs JSF.

## Découvertes importantes

1. **Les endpoints cachés des portails JSF peuvent être plus puissants que l'interface utilisateur**. Le bouton "créer PDF" de `gate.estv.admin.ch` expose en réalité un endpoint POST qui génère un PDF complet en FR — une fois la locale changée via POST. Personne n'aurait deviné ça en regardant l'interface.

2. **SPARQL Fedlex** est une ressource technique précieuse pour les lois suisses. Documentation : `https://fedlex.data.admin.ch/sparqlendpoint`. Ontologie `jolux` bien conçue. À exploiter pour toutes les futures recherches d'URL XML.

3. **Les scores RAG sur les Info TVA (0.66-0.78) sont les meilleurs obtenus à ce jour**. Raison : ces documents utilisent un vocabulaire très spécifique qui matche parfaitement les queries d'un expert-comptable. Cela suggère que **pour les tâches TVA, l'agent devrait donner priorité absolue aux Info TVA** dans le RAG plutôt que LTVA elle-même (les lois sont plus abstraites).

4. **Le parser regex du cache VS a extrait 339 articles vs 175** — presque 2× de couverture. La leçon : **toujours vérifier les statistiques de parsing** (compter les patterns détectés vs extraits) avant de déclarer un parser "OK".

5. **Postgres 14 est suffisant** pour notre usage event-sourcing — JSONB, LISTEN/NOTIFY, partitioning, tout est là. Pas besoin de migrer vers 16 tant qu'on n'atteint pas de limites spécifiques (pg_stat_monitor, compression TOAST améliorée, etc.).

6. **La KB avance plus vite que prévu** — à ce rythme, la couverture fédérale sera complète en session 06-07, et on pourra scaffold le backend et démarrer le développement applicatif bien avant T2 2026 (roadmap initiale).
