### 5. Onboarding wizard — 4 étapes

Étapes :

## Décisions structurelles prises

1. **Stack frontend figée** : React 19 + Vite 8 + TS 6 + Tailwind 3.4 + Zustand + TanStack Query. Pas de Next.js (overkill pour un SPA interne), pas de shadcn/ui (trop d'abstraction pour un scaffold initial — on écrit nos propres classes Tailwind).

2. **Proxy Vite `/api` → backend .59** : en dev, le frontend appelle `/api/*` qui est rewrité vers `http://192.168.110.59:3010/*`. En prod, `import.meta.env.DEV` false et fallback sur `VITE_LEXA_URL` (à configurer au moment du deploy).

3. **Port strict 5190** : le user a déjà plusieurs projets Vite qui tournent (ports 5173-5182 occupés). Port 5190 évite toute collision et `strictPort: true` force l'échec propre si collision.

4. **Zustand persist pour `companyStore`** : permet de rester loggé après refresh. Session 11 : ajouter vrai backend session (JWT) si on veut du multi-user.

5. **Alias `@/*` → `src/*`** : plus lisible que les `../../../` relatifs. Configuré des 2 côtés (tsconfig + vite).

6. **Types miroirs backend** : approche "copy-and-adapt" plutôt que générer depuis OpenAPI. Le backend n'a pas de schéma OpenAPI ; j'ai inspecté les shapes live au curl et écrit les types à la main. Simple, marche, à refacto si on ajoute un schéma.

7. **Framer-motion pour wizard** : transitions doucement, `AnimatePresence mode="wait"` évite le double render. Budget animation minimal.

8. **Deploy du frontend reporté session 11** : on valide d'abord localement avant de pousser sur .59. Le scaffold fonctionne, autant ne pas précipiter Nginx + cert + build rsync si on va encore itérer sur l'UI.
