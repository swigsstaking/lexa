**Nouvelle méthode** : le tail persistant vit sur `.59` au lieu de sur

**Résultat observation live** : le restart `pm2 restart lexa-backend`
du bloc C (~07:25 UTC) a cassé le stream de logs. Le tail `pm2 logs
lexa-backend` suit un process spécifique — quand pm2 tue+relance le
process child, les nouveaux logs vont au nouveau process mais le stream
déjà attaché pointe vers un PID mort. Fichier de capture : 18 lignes,
0 POST, 0 erreur.

**Leçon** : ne pas restart pm2 pendant une fenêtre d'observation cron.
Tension de design : j'avais besoin du restart pour déployer l'agent GE
du bloc C. Soit on reporte le bloc C à après le tick `:30` (donc
>30 min d'attente improductive), soit on accepte que l'observation
naturelle soit compromise et on force un test synthétique à la fin.

**Fallback synthétique exécuté à 07:35 UTC** (comme prévu par le
briefing) :

```bash
curl -s -X POST http://localhost:3010/connectors/bank/ingest \
  -H "X-Lexa-Signature: sha256=${HMAC(body, LEXA_WEBHOOK_SECRET)}" \
  -d '{"transactions":[{"txId":"synth-s16-...",...,"source":"swigs-pro-email"}],"classify":true}'
```

La boucle complète **HMAC → ingest → lexa-classifier → event store →

### Bloc A — Ingestion KB Genève

**Investigation URL (subagent Explore, ~10 min)**

Le briefing session 16 proposait `https://www.ge.ch/legislation/rsg/f/rsg_d3_05.html`.
Test `curl` → retourne la home ge.ch (titre "ge.ch", pas le contenu de
la loi). Le path a été supprimé ou n'a jamais existé en 2026.

Un subagent Explore a trouvé la source canonique : **SILGeneve**
(Service de législation de la Chancellerie cantonale GE), qui sert les
3 lois en HTML statique depuis :

- **LCP** (RSG D 3 05) → `https://silgeneve.ch/legis/data/rsg_d3_05.htm` (542 KB, à jour 2026)
- **LIPP** (RSG D 3 08) → `https://silgeneve.ch/legis/data/rsg_d3_08.htm` (278 KB, à jour 2026)
- **LIPM** (RSG D 3 15) → `https://silgeneve.ch/legis/data/rsg_d3_15.htm` (125 KB, dernière modif 2025)

Format : HTML legacy ASP.NET avec `<table>` + `<font>`, **encoding
windows-1252**. Pas besoin de Playwright — `curl -sL` + `iconv` +
BeautifulSoup suffit. Beaucoup plus stable et rapide que lexfind.ch
(qui est une SPA Angular).

**Comptage initial** via `curl | iconv | grep -oE "Art\.\s*[0-9]+[a-z]?"` :

- LCP : 327 refs Art.
- LIPP : 81 refs Art.
- LIPM : 57 refs Art.

Cohérent avec ~300-400 articles uniques après déduplication (la table
des matières en tête répète chaque Art. une fois).

**Script `01-knowledge-base/scripts/ingest_ge_laws_lexa.py`**

Clone structurel du `ingest_vs_loi_fiscale_lexa_v2.py` session 04, avec
3 différences clé :

1. **Fetch HTTP simple** via `urllib.request` avec User-Agent Chrome +
   decode windows-1252 → latin-1 fallback. Pas de cache pré-scrapé
   (comme VS v2) parce que les lois GE sont légères (< 1 MB total).
2. **Parser `parse_articles`** : pattern regex `Art. (N)(heading?)`
   avec reprise sur ligne suivante si inline vide. Filtrage des chunks
   avec body < 20 caractères (élimine la TOC). Déduplication par
   `article_num` en gardant le body le plus long.
3. **Upsert en 3 passes** (une par loi), chacune avec son tag `law` :
   `LCP-GE`, `LIPP-GE`, `LIPM-GE`. Metadata partagée :
   `jurisdiction: "cantonal-GE"`, `canton: "GE"`, `source: "silgeneve"`,
   `date_version: "2026-01-01"`.

BGE-M3 via `FlagEmbedding` (pas de passage par llama-server cette fois
car l'ingestion est un script Python sur Spark avec GPU direct, comme
les autres scripts d'ingestion). Batch 8, UUID4 strict, additif
(aucun `delete_collection`).
