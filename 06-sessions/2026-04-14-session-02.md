### 5. Recherche de l'URL XML LHID sur Fedlex

Pattern d'URL connu :
```
https://fedlex.data.admin.ch/filestore/fedlex.data.admin.ch/eli/cc/{year}/{ref}/{date}/fr/xml/fedlex-data-admin-ch-eli-cc-{year}-{ref}-{date}-fr-xml-{N}.xml
```

Pour LHID (RS 642.14), ELI path = `/cc/1991/1256_1256_1256/` (publiée au RO 1991 1256).

Boucle HTTP HEAD sur `xml-1` à `xml-10` → **`xml-1` retourne HTTP 200** au premier essai.

**URL finale** :
```
https://fedlex.data.admin.ch/filestore/fedlex.data.admin.ch/eli/cc/1991/1256_1256_1256/20250101/fr/xml/fedlex-data-admin-ch-eli-cc-1991-1256_1256_1256-20250101-fr-xml-1.xml
```

### 6. Création du script `ingest_lhid_lexa.py` sur le Spark

Copie + adaptation du parser AkomaNtoso, mais :
- LAW_SOURCES contient **uniquement** LHID
- Utilise **UUID4** pour les point IDs (pas de collision avec les points existants)
- **Pas de `delete_collection`** — upsert pur
- Affiche `points_count` avant et après ingestion pour traçabilité

**Problèmes techniques rencontrés** :
- Heredoc bash + f-string Python + escape des doubles quotes → SyntaxError (deux occurrences corrigées via `sed` puis `python3 -c` avec replacement)

## Découvertes importantes

1. **Le serveur .59 n'a pas Postgres installé.** Il faudra l'installer (apt-get install postgresql-16) en début de phase backend. Pas de blocage pour l'instant puisqu'on travaille encore sur la KB.

2. **Le port 3010 est libre** — les ports 3001-3009 sont pris par les autres apps Swigs. Notre numéro est tout trouvé.

3. **Le script `ingest_swiss_law.py` est destructif** (`delete_collection`) — il faut absolument éviter de le relancer tel quel tant que la collection contient nos ingestions additives. À chaque nouvelle ingestion, créer un script `_lexa.py` dédié en upsert pur.

4. **BGE-M3 trouve LHID à des scores 0.58-0.66** — c'est moins bon que certaines queries sur LTVA (qui atteignent 0.70+), mais c'est normal : LHID est une loi d'harmonisation, donc plus abstraite, avec moins de vocabulaire directement indexable. On peut améliorer en ajoutant des résumés ciblés LHID comme il y en a pour LTVA.

5. **L'encodage BGE-M3 est relativement lent sur le GB10** : ~9-10s par batch de 16 articles. Pour 108 articles, ça a pris ~60s. Pour des futures ingestions volumineuses (ex : 7 cantons × 500 articles = 3500), il faudra compter ~30 minutes. Pas bloquant.

6. **L'heredoc bash + f-strings Python avec nested quotes est un piège.** Leçon pour les prochaines sessions : éditer les scripts Python directement sur la machine cible (via ssh + nano/vim), ou utiliser `scp` d'un fichier local, plutôt que d'injecter via heredoc avec échappements complexes.
